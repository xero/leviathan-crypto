# Purpose & Provenance

`ctr_harness.c` is a CTR-mode vector generation tool built on top of the Serpent
reference ECB implementation in this directory (floppy1).  It was written during
a cryptographic audit of the leviathan TypeScript library to derive authoritative
Serpent-CTR test vectors for the audit's test suite.

Provenance chain:
```
  Ross Anderson floppy1 reference ECB (this directory)
    --> ctr_harness.c  (added February 2026)
          --> hardcoded CTR test vectors
                --> sources/leviathan/test/spec/09_ctr_vectors.test.ts
```

## Why floppy1 and not sources/serpent/:

sources/serpent/serpent.c is an alternative C implementation that uses NESSIE
byte ordering (big-endian per-word).  The leviathan library uses AES-submission
byte ordering -- the same convention as this floppy1 reference (bytes are
reversed before being packed as 32-bit little-endian words).  If
sources/serpent/ had been used as the ECB base, every key, counter value, and
output block would have required a full-byte-reversal conversion in the
harness -- introducing an additional transformation that could silently mask
errors.

**floppy1 was chosen because:**

1. It uses AES-submission byte ordering (identical to leviathan), so harness
   inputs and outputs can be compared to leviathan directly without any
   conversion.
2. It is the reference that produced floppy4's authoritative AES candidate
   submission test vectors -- the same vectors used as primary ground truth
   throughout the audit.  The ECB correctness of floppy1 is therefore already
   independently verified before the CTR harness is built on top of it.

## Self-attestation and trustworthiness:

No official Serpent-CTR vectors exist in any public corpus -- CTR mode was not
part of the original AES candidate submission.  The vectors in
09_ctr_vectors.test.ts are therefore self-derived: the harness uses floppy1's
ECB to implement CTR mode identically to leviathan, then the resulting outputs are
embedded in the test suite and used to verify leviathan.

This constitutes self-attestation -- leviathan is being verified against vectors
generated by code that mirrors its own construction.  What makes it
trustworthy:

1. The floppy1 ECB was independently verified against floppy4 KAT vectors
   before the CTR harness was built on top of it (see verification step 4).
2. The CTR construction in the harness is a direct, line-for-line translation
   of leviathan's CTR class in blockmode.ts -- any discrepancy between the two
   would indicate a transcription error, not a hidden implementation flaw.
3. The ECB cross-corpus sanity tests in 09_ctr_vectors.test.ts tie the CTR
   vectors back to the independently-verified ECB corpus: for all-zero
   plaintext, the first ciphertext block equals ECB_encrypt(key, IV), which
   is directly confirmed against floppy4 KAT vectors.

# Platform Notes & Portability

Discovered issues:

## 64-bit WORD on arm64 macOS

The floppy1 source uses: `typedef unsigned long WORD;`

On arm64 macOS (Apple Silicon) and x86-64 Linux, "unsigned long" is 8 bytes
(64-bit), not 4 bytes.  Therefore `BLOCK = WORD[4]` occupies 32 bytes, not 16.
Data is stored in the lower 32 bits of each 64-bit WORD slot, leaving the
upper 32 bits of each slot as zeros.

The original floppy1 code works correctly despite this because it always
manipulates a BLOCK through WORD arithmetic -- it never interprets the raw
bytes directly.  The harness, however, must supply a 16-byte counter value to
blockEncrypt and retrieve a 16-byte keystream from it. Casting a `uint8_t[16]`
pointer directly to `BYTE*` and passing it as a BLOCK produced zero-interleaved
output: e.g. `4D136B97 00000000 19DF4D0A 00000000 ...` (data in the low 4 bytes
of each 8-byte WORD slot, zeros in the high 4 bytes)

**Fix:** bytes_to_block / block_to_bytes:

The harness maintains the counter and keystream as `uint8_t[16]` (plain bytes)
and converts to/from BLOCK only for the blockEncrypt call using explicit helpers
that pack/unpack through WORD arithmetic.  The layout convention matches
`render()` in `serpent-aux.c,` which prints `w[3]` first (most significant)
through `w[0]` last (least significant), each word big-endian within its 4-byte lane:

```
blk[3] = (b[0]<<24)|(b[1]<<16)|(b[2]<<8)|b[3]     bytes  0-3  -> word 3
blk[2] = (b[4]<<24)|(b[5]<<16)|(b[6]<<8)|b[7]     bytes  4-7  -> word 2
blk[1] = (b[8]<<24)|(b[9]<<16)|(b[10]<<8)|b[11]   bytes  8-11 -> word 1
blk[0] = (b[12]<<24)|(b[13]<<16)|(b[14]<<8)|b[15] bytes 12-15 -> word 0
```

| Testing  | Platform |
| -------: | :------- |
| Platform | darwin-arm64 (Apple Silicon) |
| OS       | macOS Darwin 25.3.0 |
| Compiler | Apple clang version 17.0.0 (clang-1700.6.3.2) |
| Target   | arm64-apple-darwin25.3.0 |
| Flags    | `-Wall -O2 -Wno-unused-function -Wno-unused-but-set-variable`|

**x86-64 Linux note:**

On x86-64 Linux (LP64), "unsigned long" is also 8 bytes, so the same WORD-size
issue applies and bytes_to_block/block_to_bytes are equally necessary.  On
32-bit platforms (ILP32) "unsigned long" is 4 bytes and the helpers remain
correct (the high-word shifts evaluate to 0).


# Build Instructions

Prerequisites: `clang` or `gcc`, and `make`.
```
# Verify your compiler
clang --version
# or:
gcc --version

# Build just the CTR harness
make ctr_harness
# Produces: ./ctr_harness

# Build the full original submission suite (optional)
make all
```

On macOS, gcc is typically aliased to Apple clang.  To use a different compiler:

```
  CC=gcc make ctr_harness
```

No special flags beyond those already in the Makefile are required.  The harness
links against `serpent-aux.c` and `serpent-reference.c`, these original 1998 Ross
Anderson files and have been left unmodified.

### Output format:

| Key   | Value |
| ----- | ----- |
|  CASE | case label (A-E) |
|  KEY  | key bytes as uppercase hex (128, 192, or 256 bits) |
|  IV   | 16-byte initialisation vector as uppercase hex; loaded directly into the counter register |
|  PT   | plaintext bytes as uppercase hex (one or more 16-byte blocks) |
|  CT   | ciphertext bytes as uppercase hex (same length as PT) |

For cases with all-zero plaintext (A, B, E), CT is the raw keystream:

```
  CT_block_0 = ECB_encrypt(key, IV)
  CT_block_1 = ECB_encrypt(key, IV+1)   (counter incremented little-endian)
  etc.
```

## Verification Workflow

A future auditor wishing to independently verify the CTR test vectors:

1. Build and run the harness, capture output:

```
git checkout floppy1
make ctr_harness
./ctr_harness
```

Compare output byte-for-byte against the expected output shown above.

2. Compare against the leviathan test file:

Open `leviathan/test/spec/09_ctr_vectors.test.ts` and find the VECTORS
array.  Every CT value must match the harness output for the same case.

3. Run the full leviathan test suite:

```
bun vitest run
# For per-test output:
bun vitest run --reporter=verbose
```

All tests should pass, including the 17 in `09_ctr_vectors.test.ts`.

>[!NOTE]
> the Monte Carlo suites take approximately 4-5 minutes to complete.

4. Manual ECB cross-check (ties CTR vectors to the floppy4 ECB corpus):

Case A uses an all-zero 128-bit key, all-zero IV, and all-zero plaintext.
B/ecause plaintext is all zero, CT = raw keystream.

Keystream block 0 = ECB_encrypt(all-zero 128-bit key, all-zero counter).

Expected: `E9BA668276B81896D093A9E67AB12036`

Verify this independently via `ecb_vk.txt` (KEYSIZE=128, variable key,
zero plaintext, the all-zero-key row gives this exact ciphertext).

The automated version of this check is the "Cross-corpus ECB sanity" tests in
`09_ctr_vectors.test.ts` (for Case A with 128-bit key and Case B with 256-bit key).
